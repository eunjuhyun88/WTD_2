## Step 1. 변경 범위 파악

```bash
git diff --stat origin/main..HEAD
git diff --name-only origin/main..HEAD
```

- 변경 파일 목록과 각 파일의 역할(engine 비즈니스 로직 / app surface / 테스트 / 계약) 분류
- **engine ↔ app 경계 교차** 있으면 즉시 경고

---

## Step 2. 아키텍처 리뷰 (코드 정적 분석)

각 변경 파일에 대해 다음을 grep/read로 확인:

| 항목 | 확인 방법 | 판정 기준 |
|------|----------|----------|
| N+1 쿼리 | 루프 내 DB 호출 패턴 grep | 루프 내 `.execute()` = ❌ |
| SELECT * | `select("*")` 또는 `.select()` 인자 없음 | 컬럼 명시 없으면 ❌ |
| limit 누락 | `.execute()` 전 `.limit()` 없음 | 외부 쿼리에 limit 없으면 ❌ |
| 인증 누락 | app route에 `requireAuth` / `getAuthUserFromCookies` | 없으면 ❌ |
| 민감값 하드코딩 | secret/key 리터럴 grep | 있으면 ❌ BLOCKER |
| 외부 API 재시도 | backoff/retry 패턴 없이 단일 호출 | 없으면 ⚠️ |
| 폴백 없는 Supabase 의존 | try/except 없는 supabase 호출 | 없으면 ⚠️ |
| Pydantic/Zod 검증 누락 | 외부 입력이 스키마 통과 없이 사용 | 없으면 ❌ |

## Step 3. 실측 벤치마크 (숫자 없으면 미완)

변경된 함수/경로에 대해 **pytest 또는 inline python**으로 직접 측정:

### 3-A. Latency (ms/call)

```python
import time
for n in [small, medium, large]:   # 예: 200, 1000, 10000
    runs = 100 if n <= 1000 else 20
    start = time.perf_counter()
    for _ in range(runs): target_fn(make_input(n))
    ms = (time.perf_counter() - start) / runs * 1000
    print(f"n={n}: {ms:.3f}ms/call")
```

**통과 기준**: hot path < 5ms, cold path < 200ms

### 3-B. Memory (KB)

```python
import tracemalloc
tracemalloc.start()
s1 = tracemalloc.take_snapshot()
target_fn(large_input)
s2 = tracemalloc.take_snapshot()
kb = sum(x.size_diff for x in s2.compare_to(s1,'lineno')) / 1024
print(f"memory delta: {kb:.1f} KB")
tracemalloc.stop()
```

**통과 기준**: 단일 호출 < 10MB 증가

### 3-C. Throughput (동시 N 처리)

```python
start = time.perf_counter()
results = {k: target_fn(v) for k, v in large_batch.items()}
total_ms = (time.perf_counter() - start) * 1000
print(f"{len(large_batch)} items: {total_ms:.0f}ms | {total_ms/len(large_batch):.3f}ms/item")
```

**통과 기준**: 1000 slug/item 처리 < 5초

### 3-D. URL/payload 크기 한계 (Supabase IN clause 등)

```python
import urllib.parse, uuid
for n in [100, 200, 220, 500, 1000]:
    ids = [str(uuid.uuid4()) for _ in range(n)]
    param = "id=in.(" + ",".join(ids) + ")"
    url_len = len(urllib.parse.quote(param))
    print(f"IN({n}) → {url_len}B {'❌ OVERFLOW' if url_len > 8192 else '✅ OK'}")
```

## Step 4. 병목 식별 (실측 기반)

Step 3 숫자를 바탕으로:

- **가장 느린 경로** 1개 명시 (파일:줄번호)
- **스케일 문제** (n이 커질 때 비선형 증가 여부) — O(n) vs O(n²)
- **메모리 누수 후보** — 캐시 TTL 없이 계속 증가하는 구조

## Step 5. Import side-effect 검증

```bash
uv run python -c "
import sys, time, importlib
sys.path.insert(0, '.')
for mod in ['변경된.모듈1', '변경된.모듈2']:
    before = set(sys.modules.keys())
    t = time.perf_counter()
    importlib.import_module(mod)
    ms = (time.perf_counter()-t)*1000
    added = len(set(sys.modules.keys()) - before)
    print(f'{mod}: {ms:.0f}ms +{added} deps')
"
```

**통과 기준**: 초기 import < 500ms, 불필요한 heavy dep(ML 라이브러리 등) 자동 로드 없음

[병목]
  최대 병목: <파일:줄> — <원인>

  <모듈>: Xms +Y deps ✅/❌

[테스트]
  X passed / Y failed

[종합 판정]

- ❌ 하나라도 있으면 BLOCK
- ⚠️ 만 있으면 WARN (이슈 등록 조건)
- 모두 ✅면 PASS
