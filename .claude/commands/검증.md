## Step 1. 변경 범위 파악

```bash
git diff --stat origin/main..HEAD
git diff --name-only origin/main..HEAD
```

각 파일을 레이어로 분류:

| 파일 | 레이어 | 역할 |
|------|--------|------|
| engine/... | L1~L7 비즈니스 로직 | |
| app/src/routes/... | Surface (UI/API proxy) | |
| app/src/lib/contracts/... | 계약 경계 | |
| spec/ / docs/ | 설계 진실 | |
| tests/ | 검증 | |

## Step 2. Charter 정합성 확인 (전체 구조 기준 1)

`spec/CHARTER.md`를 읽고 변경 내용이 범위를 벗어나지 않는지 확인:

**In-Scope 체크** — 변경이 다음 7-Layer 갭 중 어디에 해당하는가:
- L3 Pattern Object / L4 State Machine / L5 Search / L6 Ledger / L7 Refinement
- AI Parser / Visualization Intent Router / LambdaRank / Semantic RAG

**Non-Goal 침범 체크** — 다음 키워드가 변경 내용에 있으면 즉시 ❌ BLOCK:
```bash
grep -rn "copy_trad\|leaderboard\|subscription\|memkraft.*new\|agent.*dispatch" \
  <변경된 파일들>
```

**결과**: In-Scope ✅ / Non-Goal 침범 ❌ / 애매함 ⚠️ (이유 명시)

## Step 3. 7-Layer 아키텍처 정합성 (전체 구조 기준 2)

`docs/domains/` 와 `spec/PRIORITIES.md`를 참조해 변경이 레이어 규칙을 지키는지 확인:

| 규칙 | 확인 방법 | 판정 |
|------|----------|------|
| `app/`은 surface만 — 비즈니스 로직 금지 | app 파일에서 DB 직접 호출, ML 계산, 통계 로직 grep | ❌ 있으면 BLOCK |
| `engine/`은 API 응답 포맷 신경 쓰지 않음 | engine에서 Svelte/TS 특화 포맷 생성 여부 | ❌ 있으면 BLOCK |
| 계층 간 통신은 OpenAPI 계약 경유 | engine ↔ app 직접 import 없는지 확인 | ❌ 있으면 BLOCK |
| L6 Ledger는 4-split 구조 준수 | entry/score/outcome/verdict 분리 여부 | ⚠️ 위반 시 주의 |
| APScheduler job은 engine 내부에만 | app/에 scheduler 코드 없는지 | ❌ 있으면 BLOCK |

## Step 4. 계약(Contract) 정합성 (전체 구조 기준 3)

```bash
# engine-openapi.d.ts가 실제 engine과 일치하는지 확인

```

계약 불일치 = ❌ BLOCK (CI에서 잡히기 전에 먼저 잡는다)

## Step 5. 코드 품질 리뷰 (정적 분석)

각 변경 파일에 대해 grep/read로 확인:

| 항목 | 확인 방법 | 판정 기준 |
|------|----------|----------|
| N+1 쿼리 | 루프 내 `.execute()` | ❌ |
| SELECT * | `.select()` 인자 없음 | ❌ |
| limit 누락 | 외부 노출 쿼리에 `.limit()` 없음 | ❌ |
| 인증 누락 | app route에 `requireAuth` / `getAuthUserFromCookies` 없음 | ❌ |
| 민감값 하드코딩 | secret/key/token 리터럴 grep | ❌ BLOCKER |
| Supabase IN clause overflow | `.in_()` 호출에 배치 분할 없이 무제한 | ❌ |
| 폴백 없는 Supabase 의존 | try/except 없는 supabase 호출 | ⚠️ |
| 외부 API 재시도 없음 | backoff 패턴 없는 단일 API 호출 | ⚠️ |
| Pydantic/Zod 스키마 우회 | 외부 입력이 검증 없이 내부 사용 | ❌ |
| 캐시 TTL 없는 인메모리 축적 | dict/list에 무제한 append | ⚠️ |

## Step 6. 실측 벤치마크 (숫자 없으면 미완)

변경된 함수/경로에 대해 **pytest 또는 inline python**으로 직접 측정:

### 6-A. Latency (ms/call)

```python
import time
for n in [small, medium, large]:   # 예: 200, 1000, 10000
    runs = 100 if n <= 1000 else 20
    start = time.perf_counter()
    for _ in range(runs): target_fn(make_input(n))
    ms = (time.perf_counter() - start) / runs * 1000
    print(f"n={n}: {ms:.3f}ms/call")
```

**통과 기준**: hot path(캐시 히트) < 5ms / cold path(DB 포함) < 200ms

### 6-B. Memory (tracemalloc)

```python
import tracemalloc
tracemalloc.start()
s1 = tracemalloc.take_snapshot()
target_fn(large_input)
s2 = tracemalloc.take_snapshot()
kb = sum(x.size_diff for x in s2.compare_to(s1,'lineno')) / 1024
print(f"memory delta: {kb:.1f} KB")
tracemalloc.stop()
```

**통과 기준**: 단일 호출 < 10MB 증가

### 6-C. Throughput

```python
start = time.perf_counter()
results = {k: target_fn(v) for k, v in large_batch.items()}
total_ms = (time.perf_counter() - start) * 1000
print(f"{len(large_batch)} items: {total_ms:.0f}ms | {total_ms/len(large_batch):.3f}ms/item")
```

**통과 기준**: 1000 item 처리 < 5초

### 6-D. 페이로드 크기 한계 (Supabase IN clause, URL length 등)

```python
import urllib.parse, uuid
for n in [100, 200, 220, 500, 1000]:
    ids = [str(uuid.uuid4()) for _ in range(n)]
    param = "id=in.(" + ",".join(ids) + ")"
    url_len = len(urllib.parse.quote(param))
    print(f"IN({n}) → {url_len}B {'❌ OVERFLOW' if url_len > 8192 else '✅ OK'}")
```

## Step 7. 병목 + 스케일 분석

Step 6 숫자 기반으로:

- **가장 느린 경로** 1개 명시 (파일:줄번호 + 원인)
- **복잡도**: O(n) / O(n log n) / O(n²) 판단
- **100명 동시 사용자 기준** 추정:
  - 동시 요청 100 × latency → 총 부하 추정
  - 캐시 적중률이 0%일 때 Supabase 요청 수 추정
- **메모리 누수 후보**: TTL 없이 계속 쌓이는 구조

## Step 8. Import side-effect + 통합 테스트

```bash
# Import 검증
uv run python -c "
import sys, time, importlib
sys.path.insert(0, '.')
for mod in ['변경된.모듈1', '변경된.모듈2']:
    before = set(sys.modules.keys())
    t = time.perf_counter()
    importlib.import_module(mod)
    ms = (time.perf_counter()-t)*1000
    added = len(set(sys.modules.keys()) - before)
    print(f'{mod}: {ms:.0f}ms +{added} deps')
"

# 테스트 실행
uv run pytest <관련 테스트 파일> -v --tb=short 2>&1 | tail -20
```

**통과 기준**: import < 500ms / 관련 테스트 전부 pass
