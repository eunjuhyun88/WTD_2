## 핵심 질문 3개 (이 순서로)

1. **"스펙이 요구한 것을 실제로 달성했는가?"** — PRD 수치 목표 + Exit Criteria
2. **"전체 시스템 안에서 올바른 위치에 있는가?"** — Charter + 7-Layer + 계약
3. **"코드가 그 목표를 올바르게 구현했는가?"** — 실측 수치 + 코드 품질

선언 금지. 모든 항목은 grep/실행/측정으로 증명.

---

## Step 1. 목표 추출 (기준선 확보)

변경 내용이 달성해야 할 목표를 3곳에서 읽는다.

```bash
# work item Exit Criteria
ls work/active/W-*.md | xargs grep -l "$(git diff --name-only origin/main..HEAD | head -3 | xargs basename -a)" 2>/dev/null
cat work/active/W-XXXX-*.md | grep -A 30 "Exit Criteria\|Goal\|성능 목표"

# PRD 수치 목표
grep -n "≥\|≤\|recall@\|latency\|p95\|accuracy\|false positive\|< [0-9]" \
  spec/PRIORITIES.md spec/CHARTER.md
```

**이 표를 반드시 채운다 — 없으면 다음 단계 금지:**

| 목표 | 출처 | 기준값 | 측정 방법 |
|------|------|--------|----------|
| (예) F-60 gate 통과 | PRIORITIES.md | verdict≥200 AND accuracy≥0.55 | 경계값 테스트 |
| (예) p95 latency | PRIORITIES.md | < 2,000ms | 실측 p95 |
| (예) recall@10 | PRIORITIES.md | ≥ 0.7 | eval set |
| ... | | | |

## Step 2. PRD 목표 달성 실측 (핵심)

Step 1 표의 각 항목을 **직접 측정**한다. 수치 추정 금지.

### 2-A. 정확도 / 정합성 목표

해당 기능의 핵심 비즈니스 로직 경계값 실행:

```python
# 예: F-60 gate — verdict 200+, accuracy ≥ 0.55
from stats.user_accuracy import _safe_div, _F60_GATE_MIN_COUNT, _F60_GATE_MIN_ACCURACY
from stats.engine import _compute_gate_status

boundary_cases = [
    # (입력, 기대값, 설명)
    (200, 0.55,  True,  "기준 딱 통과 — 반드시 PASS"),
    (200, 0.549, False, "기준 미달 — 반드시 FAIL"),
    (199, 0.60,  False, "샘플 부족 — 반드시 FAIL"),
]
for verdicts, accuracy, expected, label in boundary_cases:
    result = ...  # 실제 함수 호출
    ok = (result == expected)
    print(f"{'✅' if ok else '❌ BLOCK'} {label}")
```

### 2-B. Latency 목표 (p50/p95/p99)

```python
import time, statistics

latencies = []
for _ in range(200):
    start = time.perf_counter()
    target_fn(realistic_input)   # 실제 프로덕션 입력 크기
    latencies.append((time.perf_counter() - start) * 1000)

latencies.sort()
p50 = statistics.median(latencies)
p95 = latencies[int(len(latencies) * 0.95)]
p99 = latencies[int(len(latencies) * 0.99)]

print(f"p50: {p50:.2f}ms")
print(f"p95: {p95:.2f}ms  (PRD 목표 < 2,000ms) {'✅' if p95 < 2000 else '❌'}")
print(f"p99: {p99:.2f}ms")
```

### 2-C. 규모 목표 (100명 동시 사용자)

```python
import time

concurrent = 100
batch = {f"u{i}": realistic_input for i in range(concurrent)}

start = time.perf_counter()
results = {k: target_fn(v) for k, v in batch.items()}
total_ms = (time.perf_counter() - start) * 1000
avg_ms = total_ms / concurrent

print(f"100명 동시: {total_ms:.0f}ms 총 / {avg_ms:.1f}ms 평균")
print(f"PRD p95 < 2s: {'✅' if avg_ms < 2000 else '❌'}")
```

### 2-D. 메모리 목표

```python
import tracemalloc
tracemalloc.start()
s1 = tracemalloc.take_snapshot()
target_fn(large_realistic_input)
s2 = tracemalloc.take_snapshot()
kb = sum(x.size_diff for x in s2.compare_to(s1, 'lineno')) / 1024
print(f"메모리 증가: {kb:.0f}KB  (목표 < 10MB) {'✅' if kb < 10240 else '❌'}")
tracemalloc.stop()
```

## Step 4. 코드 품질 (구현이 올바른가)

변경 파일에 대해 grep으로 확인:

| 항목 | 명령 | 통과 기준 |
|------|------|----------|
| N+1 쿼리 | 루프 내 `.execute()` grep | 없어야 함 ❌ |
| Supabase IN clause 배치 | `.in_()` 전 분할 로직 | 100단위 배치 필요 ❌ |
| 인증 누락 | app route에 `getAuthUserFromCookies` | 없으면 ❌ |
| 민감값 하드코딩 | `secret="\|key="` 리터럴 | 있으면 ❌ BLOCKER |
| 폴백 없는 Supabase | try/except 없는 `.execute()` | ⚠️ |
| Pydantic/Zod 우회 | 외부 입력 직접 사용 | ❌ |

## Step 6. PRIORITIES.md 갭 상태 동기화

최종 보고 후 **즉시** `spec/PRIORITIES.md`를 업데이트한다. 선언 금지 — grep 실측 후 편집.

```bash
# 이번 검증 대상 work item의 feature ID 추출
grep -oE "(F|H|V|A|D|N|L|S|C)-[0-9]+" work/active/W-XXXX-*.md | sort -u

# PRIORITIES.md에서 각 ID의 현재 상태 위치 확인
grep -n "<feature_id>" spec/PRIORITIES.md
```

### ✅ PASS 판정 시

사용자 확인 후 다음을 순서대로 적용:

1. **P0/P1/P2/P3 테이블**: 해당 갭 행 삭제 (또는 `~~취소선~~`)
2. **`## 2. Architecture Reality`** 해당 Layer 갭 컬럼: `F-XX 완료 (YYYY-MM-DD)` 로 교체
3. **`## 0. CTO 현황 요약`** 헤더 3곳 갱신:
   - `열린 갭: N개 (P0=? / P1=? / P2=? / P3=?)` — 해당 priority 숫자 1 감소
   - `즉시 P0:` — 완료된 항목 제거, 다음 우선순위 항목으로 교체
   - 헤더 날짜: `| 코드 실측 기반 (커밋SHA) | YYYY-MM-DD`
4. memkraft 기록:
   ```python
   mk.log_event("gap closed: F-XX (W-XXXX, PR #NNN)", tags=["gap","priorities","f-xx"], importance="high")
   ```

### ⚠️ WARN 판정 시

- 갭 상태 유지, 해당 행에 이슈 번호 추가: `| ⚠️ WARN #NNN |`
- 갭 카운트 변경 없음

### ❌ BLOCK 판정 시

- PRIORITIES.md 수정 없음
- 갭은 미해결 상태 유지

## Step 7. Zombie Issue Sweep (항상 실행)

검증 완료 후 **항상** 다음을 실행한다. work/completed/ 에 있는 work item의 GitHub Issue가 OPEN으로 남아 있는 "좀비" 를 탐지한다.

```bash
tools/sweep_zombie_issues.sh
```

보고 예시:
```
🧟 ZOMBIE: W-0254 → #460 (work=completed, gh=OPEN)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Completed work items scanned: 211
Zombie issues found:          1
💡 실제 close: tools/sweep_zombie_issues.sh --apply
```

- **좀비 0개**: ✅ drift 없음, 계속 진행
- **좀비 1+개**: ⚠️ 사용자 확인 후 `tools/sweep_zombie_issues.sh --apply` 또는 수동 close
- BLOCK 사유 아님 — 단, PR 머지 전 정리 권장
