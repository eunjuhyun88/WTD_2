## 설계 명령어 목적

다른 에이전트가 설계문서만 읽고 **판단 없이 바로 실행**할 수 있는 수준으로 작성한다.
엔터프라이즈 소프트웨어 티켓 기준: pre-conditions / exact file ops / shell commands / verification / exit criteria 전부 포함.

## 모델 분리

| 단계 | 모델 | 이유 |
|---|---|---|

## Step 1 — 컨텍스트 수집 (병렬 실행)

```bash
# 병렬로 실행
grep -E "Frozen|Non-Goal|In-Scope" spec/CHARTER.md | head -20
grep -A3 "$ARGUMENTS" spec/PRIORITIES.md | head -20
gh issue list --search "$ARGUMENTS" --state all --limit 5
gh pr list --search "$ARGUMENTS" --state all --limit 5
git log --all --oneline --grep="$ARGUMENTS" | head -10
# 핵심 파일 LOC 실측
find app/src/routes app/src/lib -name "*.svelte" | xargs wc -l 2>/dev/null | sort -rn | head -20
ls work/active/ | grep -oE "W-[0-9]+" | sort -u | tail -5   # next W number 확인
ls app/supabase/migrations/ | tail -3
```

**Charter Frozen 항목 발견 시 즉시 중단**, 사용자에게 보고.

## Step 2 — 마스터 설계문서 작성

### 출력 파일: `work/active/W-####-{slug}.md`

```markdown
# W-#### — {Title}

> Wave: {N} | Priority: {P0/P1/P2} | Effort: {S/M/L}
> Charter: {In-Scope 근거}
> Status: 🟡 Design Draft
> Created: {YYYY-MM-DD}
> Issue: (Step 4에서 채움)
> Sub-tickets: W-####-A, W-####-B, W-####-C, ...

## Goal
사용자 입장에서 무엇이 가능해지는가 (1줄, 완료형).

## Why Now
이걸 지금 해야 하는 이유 (기술 부채 비용 / 블로커 / 기회).

## Scope
**포함:**
- 기능/동작 경계 (예: "X가 Y로 바뀐다")
- 파일 디렉토리 (실측 경로만)
- API/계약 변경 여부

**제외 (이유):**
- 제외 항목과 이유

## Architecture Decision

### 핵심 결정 (사용자 검토 필요한 것만)

| ID | 결정 | 선택 이유 | 거절 옵션 + 이유 |
|---|---|---|---|
| D-1 | ... | ... | A안: ... / B안: ... |

### 최종 디렉토리 구조 (after)
(정확한 경로, 에이전트가 mkdir 명령 그대로 쓸 수 있어야 함)

### 명명 규칙
(파일명, export 명, store 명 패턴 — 예시 포함)

### 데이터 흐름
(SvelteKit load → store / props / context — 케이스별)

### 컴포넌트 경계 규칙
(단일 책임 기준, props ≤N, 테스트 가능성 기준)

## Sub-ticket 목록

| 티켓 | 작업 | 선행 조건 | 예상 PR 크기 |
|---|---|---|---|
| W-####-A | Phase A: ... | 없음 | ~N 파일 |
| W-####-B | Phase B: ... | W-####-A merged | ~N 파일 |

## CTO 관점

### Risk Matrix
| 리스크 | 확률 | 영향 | 완화 전략 |
|---|---|---|---|

### Rollback Plan
각 Phase는 독립 revert 가능. 방법: `git revert {commit}` → redeploy.

### Files Touched (실측 기반 총계)
Phase A: N, Phase B: N, ...

## AI Researcher 관점

### 데이터 흐름 무결성
(AI/엔진 데이터가 새 구조에서 동일하게 흐르는지 — 구체 경로)

### 마이그레이션 안전성
(visual diff / e2e / type check 순서)

## Open Questions
- [ ] [Q-1] 결정권자가 답해야 할 것들만 (기술적으로 풀 수 있는 것은 본문에서 해결)

## Exit Criteria (전체 W-####)
- [ ] AC1: 수치 포함 — 예: `pnpm svelte-check` 0 errors
- [ ] AC2: 수치 포함
- [ ] CI green (lint + type + test)
- [ ] CURRENT.md main SHA 업데이트
```

## Step 3 — Phase 티켓 작성 (각 Phase별 별도 파일)

### 출력 파일: `work/active/W-####-{A|B|C|D}-{slug}.md`

각 Phase 티켓은 에이전트 1명이 **판단 없이** 실행할 수 있어야 한다.

```markdown
# W-####-{A} — Phase {A}: {Phase Title}

> Parent: W-#### | Status: 🔴 Blocked (대기) or 🟡 Ready
> Pre-conditions: (선행 merged PR 목록)
> Estimated files: N개 | Estimated time: N시간

## 이 Phase가 하는 일 (1줄)

## Pre-conditions Checklist
- [ ] {선행 작업} merged (SHA: )
- [ ] `pnpm svelte-check` 0 errors on main
- [ ] 브랜치 생성: `git checkout -b feat/W-####-A-{slug}`

## Setup Commands
```bash
# 이 명령어를 순서대로 실행
mkdir -p app/src/lib/...
```

## Operations Table (exact, ordered)
에이전트는 이 표를 위에서 아래로 순서대로 실행한다.

| # | 타입 | Source 경로 | Destination 경로 | 추가 작업 |
|---|---|---|---|---|
| 1 | MKDIR | — | app/src/lib/hubs/terminal/ | |
| 2 | MOVE | app/src/lib/cogochi/AppShell.svelte | app/src/lib/hubs/terminal/TerminalShell.svelte | import path 업데이트 |
| 3 | EDIT | app/src/routes/cogochi/+page.svelte | (동일) | import 경로 수정 |
| 4 | DELETE | app/src/lib/cogochi/ | — | 비면 삭제 |

MOVE 후 반드시: 해당 파일을 import하는 모든 파일의 경로 수정.
```bash
# 이동한 컴포넌트를 import하는 파일 찾기 (예시)
grep -rn "cogochi/AppShell" app/src --include="*.svelte" --include="*.ts"
```

## Import Path Update Map
(MOVE 작업 시 각 파일별 수정할 import 목록)

| 파일 | 기존 import | 새 import |
|---|---|---|

## Component Split Spec (Phase B 해당 시)
분할 대상 컴포넌트마다:

### {ComponentName}.svelte (현재 N LOC → N개로 분할)

**책임**: (이 컴포넌트가 하는 딱 한 가지)

**분할 후 파일:**
| 파일 | 책임 | Props 인터페이스 | 예상 LOC |
|---|---|---|---|
| FooShell.svelte | 레이아웃 조율 | `{ symbol: string; mode: Mode }` | ~150 |
| FooChart.svelte | 차트 렌더링만 | `{ payload: ChartPayload; config: ChartConfig }` | ~300 |

**분할 절차:**
1. 기존 파일 그대로 둔 채 새 파일 생성
2. 로직 이동 후 기존 파일에서 re-export
3. 테스트 후 기존 파일 삭제

## Store Fate Table (Phase C 해당 시)
각 스토어 파일의 처분:

| Store 파일 | 사용처 (grep 결과) | 처분 | 이동 위치 |
|---|---|---|---|
| activePairStore.ts | 0건 확인 후 | DELETE | — |
| priceStore.ts | routes/terminal (8건) | KEEP → MOVE | hubs/terminal/stores/ |

**삭제 전 반드시:**
```bash
# 삭제 전 사용처 0 확인 (store 파일명으로 검색)
grep -rn "{store_name}" app/src --include="*.svelte" --include="*.ts" | grep -v "lib/stores/"
# 0이면 삭제, 아니면 KEEP으로 변경
```

## Verification Commands
이 Phase 완료 후 반드시 통과해야 하는 체크:

```bash
# 1. 타입 에러 0
cd app && pnpm svelte-check 2>&1 | grep -c "^Error"
# → 반드시 0

# 2. 삭제된 트리 import 없음 (예: cogochi 이동 후)
grep -rn "from.*lib/cogochi" app/src --include="*.svelte" --include="*.ts" | wc -l
# → 반드시 0

# 3. 빌드 성공
cd app && pnpm build 2>&1 | tail -5

# 4. (Phase B) 분할 전 스크린샷과 비교
# 분할 전 baseline 저장 위치: /tmp/baseline-W-####-B/
```

## Commit & PR
```bash
git add app/src/lib/hubs/ app/src/lib/cogochi/ app/src/components/terminal/ ...
git commit -m "refactor(W-####-A): {phase title} — {N} files moved"
gh pr create \
  --title "[W-####-A] Phase A: {title}" \
  --body "## Changes\n- ...\n\nCloses part of #ISSUE_NUM\n\n## Verification\n- [ ] svelte-check 0 errors\n- [ ] Visual baseline unchanged"
```

## Exit Criteria (이 Phase)
- [ ] AC-A1: ...
- [ ] AC-A2: Verification Commands 전부 통과
- [ ] PR merged
```

## Step 4 — 사용자 검토

마스터 + 모든 Phase 티켓 초안을 보여주고 응답 대기:
- **`y`** → Step 5 (저장 + Issue 생성)
- **수정** → 섹션 + 내용 지정 후 재작성, 재검토
- **취소** → 종료 (파일 저장 안 함)

## Step 5 — 저장 + Issue 생성 (y 응답 시에만, Atomic)

Issue 생성 실패 시 파일 저장 없이 중단:

```bash
# 마스터 Issue
ISSUE_URL=$(gh issue create \
  --title "[Wave-{N}] W-#### — {Title}" \
  --body "$(cat <<'EOF'
## Goal
{1줄}

## Sub-tickets
- W-####-A: Phase A — {title}
- W-####-B: Phase B — {title}
...

## Exit Criteria
{AC 목록}

See work/active/W-####-{slug}.md
EOF
)" \
  --label "wave-{N},{priority}")
ISSUE_NUM=$(echo "$ISSUE_URL" | grep -oE '[0-9]+$')
```

파일 저장 (마스터 + 각 Phase):
- `work/active/W-####-{slug}.md` (Issue 번호 frontmatter에 추가)
- `work/active/W-####-A-{slug}.md`
- `work/active/W-####-B-{slug}.md`
- ...

map 등록:
```bash
tools/work_issue_map.sh add W-#### "$ISSUE_NUM" --status design
```

보고: 파일 경로 목록 + Issue URL

## 필수 규칙

- **3+ 파일 / 새 시스템 / 구조 변경 = 이 명령어 필수**
- **단순 1파일 수정은 생략 가능**
- TBD 금지 — 모르면 Open Question
- Exit Criteria 수치 없으면 반려
- 거절 옵션은 이유 포함
- LOC 상한 없음 — "단일 책임 + Props ≤8 + 독립 테스트 가능"이 분할 기준
- Phase 티켓은 에이전트가 판단 없이 실행 가능해야 함 (모호한 지시 금지)
