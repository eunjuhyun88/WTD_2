설계 대상: $ARGUMENTS

## 원칙

**Charter 준수**: `spec/CHARTER.md` In-Scope (L3~L7) 위반 시 즉시 중단.
Frozen: copy_trading, leaderboard, AI 차트 분석, 자동매매, 신규 메모리 stack.

**Design-First Loop**: 코드 한 줄 쓰기 전 설계문서 먼저. 사용자 검토 통과 후 구현.

## 단계

### Step 1 — 컨텍스트 수집 (자동, 병렬)

다음을 동시에 실행:
- `spec/PRIORITIES.md` 에서 대상 항목의 P0/P1/Wave/Effort/Decision ID 확인
- `spec/CHARTER.md` In-Scope vs Frozen 체크
- `docs/live/W-0220-product-prd-master.md` 에서 D-결정/Q-질문 확인
- `docs/live/W-0220-status-checklist.md` 에서 미완 sub-task 추출
- `git log --all --oneline --grep="<keyword>"` 최근 관련 커밋
- `gh issue list --search "<keyword>"` 기존 이슈 중복 체크
- `gh pr list --search "<keyword>"` 진행 중인 PR 중복 체크
- 관련 코드 파일 (해당하면) 줄 수 + 주요 함수 그래프

### Step 2 — 2-perspective 설계 초안

다음 구조로 `work/active/W-####-{slug}.md` 초안 작성:

```markdown
# W-#### — {Title}

> Wave: {1/2/3/4/MM/...}  | Priority: {P0/P1/P2}  | Effort: {S/M/L}
> Charter: {In-Scope L#  또는 Non-Goal 거부}
> Status: 🟡 Design Draft (사용자 검토 대기)
> Created: YYYY-MM-DD by Agent A###

## Goal (1줄)
사용자 입장에서 무엇이 가능해지는가.

## Scope
- 포함: ...
- 파일/모듈: `engine/...`, `app/...`
- API surface: `POST /...`

## Non-Goals
- 포함하지 않는 것 (왜 안 하는지 1줄)

## CTO 관점 (Engineering)

### Risk Matrix
| 리스크 | 확률 | 영향 | 완화 |
|---|---|---|---|

### Dependencies
- 선행: W-#### / PR #### (있으면)
- 차단: 어떤 다른 작업이 막히는지

### Rollback Plan
- 어떻게 되돌릴 수 있는가

### Files Touched (예상)
- `engine/...:NNN줄` — what changes
- `app/...:NNN줄` — what changes

## AI Researcher 관점 (Data/Model)

### Data Impact
- 라벨/스키마 변경 여부
- 기존 데이터 마이그레이션 필요 여부
- LightGBM/Layer-C 학습 신호 영향

### Statistical Validation
- 측정 지표 + threshold (recall@10 / NDCG@5 / accuracy 등)
- A/B 또는 holdout 계획

### Failure Modes
- 통계적 / 라벨 / drift 관점에서 무엇이 잘못될 수 있나

## Decisions (이 설계에서 확정한 것)

- [D-####] 옵션 X 선택 (옵션 Y 거절: 이유)

## Open Questions (사용자 결정 필요)

- [ ] [Q-####] ... ?

## Implementation Plan

1. {1단계 — 코드/마이그레이션}
2. {2단계}
3. {3단계 — 테스트}

## Exit Criteria

- [ ] AC1: ...
- [ ] AC2: ...
- [ ] CI green
- [ ] PR merged + CURRENT.md SHA 업데이트

## References

- spec/PRIORITIES.md §{section}
- docs/live/W-0220-product-prd-master.md §{decision}
- 기존 PR/Issue: #####
```

### Step 3 — 사용자 검토

사용자에게 보여주고 다음 중 선택:
- `y` — 그대로 저장 + Step 4 진행
- `수정` — 어떤 섹션 수정할지 받고 재작성
- `취소` — 저장 안 함

### Step 4 — 저장 + GitHub Issue 생성

확정 시:

1. 파일 저장: `work/active/W-####-{slug}.md`

2. GitHub Issue 초안 생성:
```bash
gh issue create \
  --title "[Wave-{N}/{Track}] W-#### — {Title}" \
  --body "<설계문서 §Goal + §Scope + §Exit Criteria 발췌 + 'See work/active/W-####-{slug}.md for full design'>" \
  --label "{wave-N},{priority},{track}"
```

이슈 번호 받기 전 사용자에게 한 번 더 확인:
"이슈 생성하고 main에 work/active/{file} 푸시할까? [y/n]"

3. memkraft decision 등록 (선택한 옵션이 있으면):
```bash
./tools/mk.sh decision "W-####: {one-line summary}"
```

4. 보고:
- 파일 경로
- Issue URL
- 다음 액션 (mutex 획득: `gh issue edit N --add-assignee @me`)

## 필수 규칙

- **Charter Frozen 진입 시 즉시 중단**, 사용자에게 "이건 Frozen 항목이다" 보고하고 종료
- **거절된 옵션은 거절 이유와 함께 기록** (다음 에이전트가 같은 옵션 재검토 방지)
- **TBD/추후확인 금지** — 모르면 Open Questions로
- **3+ 파일 수정 / 새 시스템 / 정책 변경 / 인프라 변경** = 무조건 이 명령어 거쳐야 함 (코드 직접 작성 금지)
- 단순 작업(파일 1개 수정, 정해진 명령 실행)은 이 명령어 생략 가능
