세션 종료 절차 (인계 + 정리 + 다음 설계 강제):

1. 이번 세션의 활동을 자동 추출:
   - `git log` 이번 세션 동안의 커밋
   - 생성/머지된 PR 목록
   - 변경된 파일 (`git diff --stat`)
   - 대화 맥락에서 "한 일 / 안 한 일 / 막힌 곳" 식별

2. `/인계` 슬래시 명령어를 호출한 것처럼 다음 구조로 초안 작성:
   - 한 일 (구체적 PR 번호/커밋 해시 포함)
   - 안 한 일 (의도적으로 스코프 외였던 것)
   - 막힌 곳 (블로커, 알 수 없었던 것)
   - 즉시 다음 단계 (다음 에이전트가 첫 5분 안에 할 일)
   - 전제 가정 (main SHA, 의존 PR 상태, 환경 — 깨지면 무효)

3. 사용자에게 초안 보여주고 [y/수정/취소] 확인. 수정 요청 시 한 줄씩 반영.

4. **머지된 work item 자동 정리** (AGENTS.md:192 enforcement):

   이번 세션 또는 origin/main에서 머지된 PR을 식별하고 매핑된 W-XXXX work item을
   `work/completed/`로 자동 이동한다.

   절차:
   - `git log --merges --since="<session_start>" --pretty="%H %s"` 또는
     `git log origin/main --oneline -20` 검토
   - 각 머지 commit/PR에서 정규식 `W-\d{4}` 추출
   - 매칭된 `work/active/W-XXXX-*.md` 표시
   - 사용자 확인 후 각각 `bash tools/complete_work_item.sh W-XXXX` 호출
   - Exit Criteria 미충족이면 `--force` 옵션 안내 (강제 완료)
   - 결과: `git mv` + `CURRENT.md` 활성표 행 제거 + memkraft log

   대상이 없으면 스킵.

4-B. **PRIORITIES.md 갭 닫기** (Step 4와 연동):

   완료된 각 W-XXXX work item에서 feature ID를 추출하고, `/검증` PASS 판정이 있으면
   `spec/PRIORITIES.md`를 즉시 업데이트한다. **선언 금지 — grep 실측 후 편집.**

   ```bash
   # 1. work item에서 feature ID 추출 (F-/H-/V-/A-/D-/S-/C- 패턴)
   grep -oE "(F|H|V|A|D|N|L|S|C)-[0-9]+" work/active/W-XXXX-*.md | sort -u

   # 2. PRIORITIES.md에서 해당 ID 위치 확인
   grep -n "F-02\|H-07\|V-00" spec/PRIORITIES.md   # 실제 ID로 교체
   ```

   **편집 규칙 (사용자 확인 후 적용):**
   - P0/P1/P2 테이블에서 해당 행: 삭제 또는 맨 앞에 `~~` 취소선
   - `## 2. Architecture Reality` 해당 Layer 갭 컬럼: `— (완료)` 로 교체
   - `## 0. CTO 현황 요약` `열린 갭: N개` 숫자 1 감소
   - `즉시 P0:` 라인: 완료된 항목 제거, 다음 P0 항목으로 교체
   - 수정 완료 날짜를 `> **CTO + AI Researcher Edition** | ...` 헤더에 반영

   `/검증` PASS 없이 머지만 된 경우: 갭 상태를 `🟡` (부분 완료)로만 변경.
   대상 feature ID가 없으면 스킵.

5. **Sweep + Drift Check** (자동 정리 + 경고):

   `tools/end.sh`가 실행 중 자동으로 다음을 호출한다 (수동 호출 불필요):
   - `tools/sweep_session_artifacts.sh` — D-1 세션 아티팩트 archive, D-2 docs/live 중복 삭제
   - `tools/check_drift.sh` — E-1 main SHA, E-2 W-#### 충돌, E-3 활성표 모순, E-4 origin/main 차이

   결과 보고를 사용자에게 전달.

6. **CTO + AI Researcher 관점으로 다음 work item 설계** (종료 전 필수):

   `tools/check_drift.sh` 의 E-2 결과(다음 사용 가능 W-#### 번호)를 반영해서 충돌 없는 번호로 생성.

   `spec/PRIORITIES.md`의 P0/P1/P2 중 `work/active/W-XXXX-*.md` 파일이 없는 항목을 찾아
   각각 `work/active/W-XXXX-<slug>.md`를 생성한다.

   **설계문서는 "선언"이 아니라 "실측 기반 설계"여야 한다.**
   작성 전에 반드시 아래 순서를 따른다:

   ---

   ### 0단계: 코드 현황 실측 (작성 전 필수)

   설계문서를 쓰기 전에 관련 코드를 직접 읽어 Facts를 채운다. 추측 금지.

   ```bash
   # 관련 파일 현황 파악
   grep -rn "<기능 키워드>" engine/ app/src/ --include="*.py" --include="*.ts" --include="*.svelte"

   # 현재 DB 스키마/마이그레이션 확인
   ls app/supabase/migrations/ | tail -5
   cat app/supabase/migrations/<최신>.sql | head -30

   # 관련 엔드포인트 존재 여부
   grep -n "@router\." engine/api/routes/<관련파일>.py

   # 현재 테스트 커버리지
   ls engine/tests/test_<관련>.py
   ```

   이 결과를 Facts 섹션에 그대로 기록한다.

   ---

   ### 1단계: PRD 수치 목표 추출

   `spec/PRIORITIES.md`와 `spec/CHARTER.md`에서 이 work item에 해당하는
   **측정 가능한 수치 목표**를 찾아 Exit Criteria에 반영한다.

   ```bash
   grep -n "≥\|≤\|recall@\|latency\|p95\|accuracy\|false positive\|< [0-9]\|WAA" \
     spec/PRIORITIES.md spec/CHARTER.md
   ```

   Exit Criteria는 반드시 수치 기준을 포함해야 한다:
   - ❌ 나쁜 예: "gate가 올바르게 작동한다"
   - ✅ 좋은 예: "verdict 200건 + accuracy ≥ 0.55 시 gate_passed=True, p95 latency < 200ms"

   ---

   ### 2단계: AI Researcher 관점 리스크 분석

   다음 항목을 설계문서의 `## AI Researcher 리스크` 섹션에 반드시 포함:

   **훈련 데이터 영향**
   - 이 변경이 LightGBM Layer C 훈련 데이터에 영향을 주는가?
   - 라벨 분포가 바뀌는가? (near_miss/too_early 비율 변화 등)
   - 변경 전 데이터와 변경 후 데이터의 schema 호환성

   **통계적 유효성**
   - 핵심 임계값(예: 0.55 accuracy)의 통계적 근거가 있는가?
   - 샘플 수가 충분한가? (95% CI 기준 최소 n 계산)
   - 자기선택 편향(self-selection bias) 리스크가 있는가?

   **실데이터 검증 시나리오**
   - 53 PatternObject corpus 기준 예상 동작
   - 138,915 feature_window rows 기준 쿼리 부하 추정
   - 실 사용자 행동 패턴 대비 edge case

   ---

   ### 3단계: CTO 설계 원칙 적용

   **성능 (100명+ 동시 사용자 기준)**
   - DB 쿼리: N+1 금지 — 루프 내 `.execute()` 패턴 설계에 없어야 함
   - Supabase: `select("col1, col2")` 컬럼 명시, limit 항상 지정
   - IN clause: 배치 100단위 분할 (URL 8KB 한계 — 220 UUID 초과 시 overflow)
   - 캐시: hot path에 TTL 명시 (현재 stats engine = 5분)
   - 비동기: blocking I/O는 `asyncio.to_thread()` 래핑 여부 판단

   **안정성**
   - 폴백: Supabase 장애 시 file fallback 또는 graceful degradation 명시
   - 멱등성: upsert conflict key 명시
   - 재시도: 외부 API 호출에 exponential backoff 명시
   - 헬스체크: 신규 경로마다 관측 가능성 포함

   **보안**
   - JWT: 모든 app API route에 `requireAuth()` / `getAuthUserFromCookies` 명시
   - RLS: Supabase 신규 테이블에 Row Level Security 정책 명시
   - 민감값: secret은 env var (`.env.example`에 키 이름 추가)
   - 입력 검증: 외부 입력 → zod(app) 또는 pydantic(engine) 스키마 통과 필수

   **유지보수성**
   - 계층 준수: `app/`은 surface only, `engine/`은 business logic — 교차 금지
   - 계약: 신규 engine route → `npm run contract:sync:engine-types` 실행 명시
   - 테스트: 핵심 경로마다 pytest 통합 테스트 1개 이상 (synthetic data 아닌 경계값 기반)
   - 롤백: migration은 non-destructive 전략 또는 down script 명시

   ### work item 필수 섹션 (이 순서로, 각 섹션 실측 기반)

   ```
   ## Goal              — 1문장, 사용자 가치 중심 (Jin 페르소나 기준)
   ## Owner             — engine / app / engine+app
   ## Scope             — 변경 파일 목록 + 각 파일 변경 이유 (코드 현황 실측 후 작성)
   ## Non-Goals         — 이번 work item에서 명시적으로 제외
   ## Exit Criteria     — PRD 수치 목표 포함 (측정 방법 명시, 수치 없으면 반려)
   ## AI Researcher 리스크 — 훈련 데이터 영향 + 통계 유효성 + 실데이터 시나리오
   ## CTO 설계 결정     — 성능/안정성/보안/유지보수성 핵심 결정 사항
   ## Facts             — grep/read 실측 결과 (추측 금지)
   ## Assumptions       — 환경 가정 (env var, migration 실행 여부 등)
   ## Canonical Files   — 건드릴 파일 전체 목록 + 각 역할
   ```
