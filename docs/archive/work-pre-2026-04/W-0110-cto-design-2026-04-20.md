### Decision 1: Pattern #13 = Liquidity Sweep Reversal

**Candidate comparison:**
- **Liquidity Sweep Reversal** (RECOMMENDED)
  - Detects: `sweep_above_high + volume_spike + quick_reversal_candle`
  - Data: Uses existing `breakout_above_high` block, `volume_spike`
  - Crypto-native: Market makers sweep stops then reverse (high-value setup)
  - Blocks needed: `sweep_detection` (new, ~50 lines), `reversal_candle` (borrowed from WSR)

- **VWAP Reclaim** (backup)
  - Simpler, proven intraday edge
  - Requires VWAP calculation block (~40 lines, new)
  - Less crypto-native; more traditional finance flavor
  - Lower upside (fewer high-conviction setups per day)

- **CME Gap Fill** (defer)
  - Requires CME futures data (blocked on COT parser, W-0111+)
  - Institutional but data-gated
  - Candidate for W-0111 after paradigm framework matures

**Rationale:**

- High crypto market relevance (stop sweeps are real phenomenon)
- Can promote via paradigm framework (multi-period, multi-model scoring)
- Unblocks W-0110-A execution within one week

### Decision 3: Candle Continuity Bug Root Cause + Fix

**Symptom:** API returns 500 bars; LWC displays only 6-8 visible on screen, but `fitContent()` should show all.

**Root cause analysis:**
1. **Timestamp unit mismatch** (LIKELY): LWC expects unix **seconds**, not milliseconds
   - Symptom: If klines return ms, LWC interprets them as future dates, skips rendering old bars
   - Check: `/api/chart/klines` response timestamp units

2. **fitContent() timing** (MEDIUM): Called on every data update, may reset zoom after user pans
   - Current: `onUpdate()` → `fitContent()` → re-render
   - Issue: User pans to recent data, then chart resets to fit all 500 bars again
   - Symptom: Jittering scroll position, bars "jump" into view

3. **Data gap detection** (LOW): LWC's timeScale may skip rendering if gaps are detected
   - Less likely; LWC handles gaps gracefully
   - But verify: are there 4h candles with missing bars due to exchange downtime?

**Fix strategy (W-0110-B, Phase 1):**

```javascript
// 1. Verify timestamp units in /api/chart/klines
// Expected: time: 1713619200 (seconds, not 1713619200000)
const klines = await fetch('/api/chart/klines?tf=4h&limit=500');
// Check: klines[0].time should be recent unix seconds (~1.7e9)

// 2. Fix fitContent() call
let isFirstLoad = true;
function onKlinesUpdate(newKlines) {
  chart.addCandleSeries(newKlines);
  if (isFirstLoad) {
    timeScale.fitContent();
    isFirstLoad = false;
  }
  // Don't call fitContent() again until user explicitly resets
}

// 3. Add scrollToRealTime() after initial fit
timeScale.fitContent();
timeScale.scrollToRealTime(); // LWC method: aligns to now
```

**Verification:**
- Render test: load 500-bar kline, verify all bars are rendered (count pixels)
- Interaction test: pan left, verify bars don't jump back to fit
- Live test: stream 4h candles, verify no jitter

**Follow-up (W-0110-C):** If timestamp issue is root cause, backfill `/api/chart/klines` unit conversion across all timeframes.

## CTO Reasoning & Architecture Notes

### Why Pattern #13 = Liquidity Sweep Reversal?

Crypto markets are dominated by **stop hunts** — coordinated sweeps of limit orders above resistance before reversals. Traditional finance doesn't see this as much (less fragmentation). This pattern is:
- **Data-independent**: No new API or external source needed
- **High-conviction**: Market makers leaving traces (visible on-chain with mev-inspect, but we detect it from price action)
- **Complementary to existing 12**: WHALE + WSR catch reversals after accumulation; Sweep catches reversals after liquidity runs

The paradigm framework already supports this (multi-model, multi-period scoring). It's a natural extension, not a rewrite.

**Alternative considered (VWAP Reclaim):** Good edge, but it's "traditional playbook." We're building crypto-native. Save VWAP for W-0114 when we want to bridge traditional traders.

### Why ChartBoard Split Now?

Performance ceiling is real. W-0102/W-0103 added more UI state to manage, and the 2300-line file is now a bottleneck:
1. **Cognitive load**: Next engineer onboarding takes 2 hours just to find "where's the candle series code?"
2. **Render perf**: Every indicator update triggers a full ChartBoard re-render (Svelte reactivity, hard to isolate)
3. **Bug isolation**: Candle continuity bug could be in 4 places; split reduces blast radius

The split also unblocks **W-0116 (Dashboard Verdict Inbox)**, which will need toolbar + indicator panels in a sidebar context. Once ChartBoard is modular, we can reuse pieces.

### Why Terminal UI Debt?

Consistency is a silent killer in trading UX. If symbol changes in SymbolPicker but doesn't update the Header pill, users blame the app, not their own perception. We're at 12 patterns; next 8 will add complexity. Before we scale up, we need **boring infrastructure** — single source of truth, no race conditions.
